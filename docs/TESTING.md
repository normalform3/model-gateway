# 测试与压测方案

## 测试原则

- 优先使用 Mock Provider 覆盖主链路，避免依赖真实模型费用和外部稳定性。
- 限流、额度、账单和消息消费必须覆盖并发与失败路径。
- SSE 流式代理必须用真实流式行为验证，不能只测普通 JSON 响应。
- 账单相关验证必须包含重复消费和补偿场景。

## Mock Provider

Mock Provider 是 MVP 必备能力。

它需要支持：

- 正常普通响应。
- 正常 SSE 流式响应。
- 慢首 Token。
- 慢完整响应。
- Provider 429。
- Provider 500。
- 上游超时。
- 流式中断。
- Usage 缺失。
- 固定 Token 数返回。
- 高并发请求。

Mock Provider 应允许通过请求参数或测试配置控制行为，便于自动化测试。

## 单元测试

优先覆盖：

- API Key 哈希和前缀生成。
- ApiKeyContext 权限判断。
- 真实模型目录解析与 Provider 凭据选择。
- Token 估算。
- 错误码映射。
- Provider 错误可重试分类。
- Usage Event 构建。
- 账单金额计算。
- 草稿团队不能申请或接收团队授权。
- 团队负责人只能给成员划拨已获批模型和团队公共池中可用的 Token。
- 成员首次有效授权时仅系统可生成一把成员应用 Key；后续授权不重复泄露明文。

## 集成测试

建议使用本地依赖或容器化依赖覆盖：

- MySQL 表结构和唯一约束。
- Redis Lua 限流脚本。
- Redis 额度冻结脚本。
- RocketMQ 消息发送和消费。
- 消费幂等事务。

关键场景：

- Key 有效、禁用、过期、无权限。
- 额度足够、额度不足、冻结后释放、冻结后结算。
- 消费者重复收到同一事件。
- 账单消费者处理失败后重试。

## 压测一：鉴权与限流吞吐

目标：不调用真实模型，使用 Mock Provider 立即返回，验证网关主链路吞吐。

验证指标：

- API Key 缓存命中率。
- Redis 限流吞吐量。
- P95 和 P99 延迟。
- 限流准确率。
- 多实例限流一致性。
- MySQL 访问次数是否符合预期。

通过标准：

- 高频请求下不会每次访问 MySQL。
- 限流结果与配置一致。
- 失败响应稳定返回标准错误码。

## 压测二：SSE 长连接

目标：Mock Provider 每固定间隔返回一个 Token，验证大量长连接下的资源占用。

验证指标：

- 最大稳定连接数。
- 首 Token 延迟。
- 完整请求耗时。
- 内存占用。
- 客户端中断后的资源释放。
- 并发计数归还准确性。

通过标准：

- 流式响应逐片返回。
- 客户端取消后，网关取消上游请求或停止继续转发。
- 并发计数不会泄露。
- 请求结束后能正确生成 Usage Event。

## 压测三：MQ 积压恢复

目标：暂停账单消费者，持续发送用量消息，然后恢复消费者。

验证指标：

- 消息积压量。
- 恢复消费速度。
- 重复消费数量。
- 消费失败重试次数。
- 死信数量。
- 最终账单准确性。

通过标准：

- 消费者恢复后能追上积压。
- 重复消息不会生成重复账单。
- 永久业务异常不会无限重试阻塞队列。

## 额度并发测试

场景：

```text
成员个人剩余额度 1000 tokens
同时发起 10 个请求
每个请求预估 200 tokens
```

期望：

- 最多 5 个请求成功冻结。
- 其余请求返回 `QUOTA_INSUFFICIENT`。
- 成功请求完成后按实际 Token 结算。
- 失败请求释放全部冻结额度。

## 分层授权与额度测试

场景：

```text
无负责人的草稿团队
负责人提交团队申请
管理员批准 1000 Token 和一个模型
负责人给成员划拨 600 Token 和该模型
```

期望：

- 草稿团队不能被管理员直接充值或授予模型。
- 审批后团队公共池为 1000 Token；划拨后为 400 Token，成员账户为 600 Token。
- 调用只冻结成员账户；团队实际账单等于该团队成员账单明细之和。
- 撤销团队或成员模型权限后，缓存命中 Key 也立即收到 `MODEL_NOT_ALLOWED`。

## 幂等测试

场景：

- 同一个客户端请求使用相同 `Idempotency-Key` 重试。
- 同一个 `UsageReportedEvent` 被 RocketMQ 重复投递。
- 账单消费者在数据库提交前失败。
- 账单消费者在数据库提交后 ACK 前失败。

期望：

- 请求记录不重复。
- 用量记录不重复。
- 账单记录不重复。
- 额度流水不重复。
- 重试后最终状态一致。

## 敏感信息检查

每次提交前检查：

```text
真实 Provider Key
私有服务端点
Workspace ID
租户 ID
签名 URL
私有 bucket 路径
部署回调地址
```

示例命令：

```bash
rg -n "signed|signature|workspace|tenant|bucket|callback|private|secret|password|api[_-]?key|provider-api-key-placeholder|mg-key-example" .
```

命中占位符是允许的；命中真实值必须移除。
