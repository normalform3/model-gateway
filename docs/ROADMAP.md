# 开发路线图

## 阶段一：模型调用闭环

目标：先完成最小可用的模型代理能力。

- 建立 Maven 多模块工程骨架。
- 定义统一 Provider 接口。
- 实现 Mock Provider。
- 实现 OpenAI-Compatible Provider。
- 提供 `/v1/chat/completions`。
- 支持普通响应。
- 支持 SSE 流式响应。
- 记录基础请求日志。

验收：

- 使用 Mock Provider 可以完成普通和流式请求。
- 流式响应不能等待完整上游响应后一次性返回。
- Mock Provider 可模拟正常、慢响应、429、500、流式中断和 Usage 缺失。

## 阶段二：虚拟 Key 与鉴权缓存

目标：业务应用不直接持有真实 Provider 凭据。

- 创建组织、团队、应用和虚拟 Key。
- 虚拟 Key 只在创建时展示一次。
- 保存 Key 前缀和 Key 哈希。
- 实现 ApiKeyContext。
- 实现 Caffeine、Redis、MySQL 三级读取。
- 实现真实模型访问权限校验。
- 实现禁用 Key 后的缓存失效。

验收：

- 有效 Key 可以访问允许的真实模型。
- 禁用、过期或无权限 Key 被拒绝。
- 数据面缓存命中时不访问 MySQL。

## 阶段三：限流、并发和额度

目标：高并发场景下稳定控制用量。

- 实现 Redis RPM 限流。
- 实现 Redis TPM 限流。
- 实现 Redis ZSET 并发控制。
- 实现 Token 估算。
- 实现额度冻结、结算和释放。
- 处理请求失败、超时和客户端取消。

验收：

- 多实例并发下不会突破 RPM、TPM 和并发限制。
- 并发请求不会造成额度超扣。
- 请求失败会释放冻结额度。
- 客户端取消后并发计数会释放，并按实际生成情况处理额度。

## 阶段四：RocketMQ 计量和账单

目标：将统计和账单从主链路解耦。

- 定义 UsageReportedEvent。
- 发送 `AI_USAGE_EVENT`。
- 实现用量消费者。
- 实现账单消费者。
- 实现 `mq_consume_record`。
- 实现账单业务唯一约束。
- 实现每日用量汇总。

验收：

- 成功、失败、取消请求都能形成事件。
- 重复消费不会产生重复账单。
- 暂停消费者后恢复，消息可继续消费并保持账单正确。

## 阶段五：模型路由和故障降级

目标：提高多模型可用性和成本控制能力。

- 实现 OpenAI-Compatible Provider 多凭据轮询与首包前故障切换。
- 记录部署并发、延迟和错误率。
- 实现简单失败重试。
- 实现部署健康状态。
- 引入熔断、半开探测和跨模型降级。
- 增加成本优先和最低负载路由。

验收：

- 权重路由比例在样本量足够时接近配置。
- 不可重试错误不会跨模型重试。
- 可重试错误最多重试有限次数。
- 熔断后请求不会继续打到故障部署。

## 阶段六：监控、压测和演练

目标：让项目具备可展示、可解释的高并发和可靠性证据。

- 接入 Micrometer 和 Prometheus 指标。
- 增加 TTFT、总延迟和网关开销指标。
- 增加 Redis 限流命中率指标。
- 增加 MQ 积压和消费失败指标。
- 完成鉴权与限流吞吐压测。
- 完成 SSE 长连接压测。
- 完成 MQ 积压恢复演练。

验收：

- 能展示 P95、P99、错误率、限流率、缓存命中率。
- 能说明性能瓶颈在网关、Redis、Provider 还是 MQ。
- 能证明消费者重复、失败、积压后账单仍可恢复。
