# ModelGate 项目协作规则

## 项目定位

ModelGate 是企业级 AI 网关与用量计费平台，运行在内部 Agent、AI 应用、研发工具和模型供应商之间。

它不是负责回答问题的 Agent，而是基础设施平台，重点解决：

- 统一大模型调用入口
- 虚拟 API Key 管理
- 多模型接入与逻辑模型路由
- Redis 分布式限流与额度控制
- SSE 流式转发
- RocketMQ 异步用量计量
- MySQL 最终账本与账单追溯

## 工作方式

- 修改代码前先读 `README.md`、`docs/PRD.md`、`docs/ARCHITECTURE.md` 和相关专题文档。
- 每次只做一个清晰任务，优先完成可验证的小闭环。
- 不要一次性重构多个模块，也不要在文档任务中顺手创建工程代码。
- 不引入新生产依赖，除非任务明确需要并已经说明原因。
- 对外接口、数据库结构、消息事件、Redis Key 规则发生变化时，必须同步更新文档。

## 架构边界

- 第一版采用 Maven 多模块的模块化单体，而不是一开始拆成多个微服务。
- 控制面负责低频管理操作，数据面负责高频模型调用链路，异步计量负责用量、账单和审计。
- 数据面主链路尽量使用本地缓存、Redis 和异步消息，避免每次请求访问 MySQL。
- Redis 是实时判断和缓存层，MySQL 是最终事实账本。
- RocketMQ 用于领域事件传播，不作为普通日志替代品。

## AI 与确定性逻辑

- 鉴权、权限、限流、额度冻结、计费、路由规则优先用确定性代码实现。
- LLM 只用于真正需要语言理解、生成、分类或模糊判断的场景。
- 不要为了显得像 Agent 系统而引入多 Agent 架构。

## 安全规则

- 仓库中不得出现真实 Provider API Key、私有服务端点、Workspace ID、租户 ID、签名 URL、私有 bucket 路径或部署回调地址。
- 文档、测试、fixture 和示例配置只能使用占位符，例如 `provider-api-key-placeholder`、`https://provider.example.com/v1`。
- 虚拟 API Key 示例只能使用明显不可用的占位符，例如 `mg-key-example`。
- 数据库中不保存明文虚拟 Key；设计上只允许保存 Key 前缀、Key 哈希和必要的加密材料。
- 不要记录完整 Prompt、完整响应或敏感业务内容到调用日志，除非后续任务明确设计了隐私策略。

## 后端规则

- WebFlux 网关链路不得阻塞等待完整上游响应后再返回流式结果。
- Redis Lua 脚本用于多维限流、并发控制和额度冻结时，必须保证原子性和失败不部分扣减。
- 额度变更必须有流水，不能只维护余额字段。
- RocketMQ 消费者必须幂等，账单相关表必须有业务唯一约束。
- 对外错误响应要可诊断，但不能泄露真实 Provider 密钥、私有端点或内部堆栈。

## 前端规则

- 管理控制台服务于平台管理员、团队管理员和开发者，优先做清晰、密集、可扫描的运维工作台。
- 不要做营销式首页；第一屏应服务于实际管理和排障。
- 所有列表和详情页要处理加载、空状态、错误状态和权限不足状态。

## 验证要求

- 文档变更至少检查 Markdown 文件清单、交叉引用和敏感信息模式。
- 代码变更优先运行最小相关测试、编译或构建。
- 网关链路变更需要覆盖普通响应和 SSE 流式响应。
- 限流、额度、账单、消息消费变更需要包含并发、失败和重复消费场景。

## 默认完成格式

完成任务时说明：

1. Summary
2. Files changed
3. Verification
4. Risks or follow-ups
5. Suggested commit message
